# ADR-004: Materialized Path for Category Tree

**Status:** Accepted
**Date:** 2025-09-22
**Deciders:** Engineering team
**Supersedes:** None

## Context

The product catalog requires a hierarchical category system with unlimited nesting depth. Common operations include:

- Display a category tree for navigation menus
- List all products in a category **and its descendants**
- Generate breadcrumb trails for product pages
- Reorder categories within a parent group
- Move a subtree from one parent to another

We evaluated three common approaches for storing hierarchical data in a relational database.

## Options Considered

### Option 1: Adjacency List

The simplest model — each category has a `parentId` foreign key.

```sql
CREATE TABLE categories (
  id VARCHAR PRIMARY KEY,
  name VARCHAR NOT NULL,
  parent_id VARCHAR REFERENCES categories(id)
);
```

**Pros:** Simple schema, easy inserts and moves
**Cons:** Recursive queries for ancestors/descendants (slow without CTEs), no single-query breadcrumbs

### Option 2: Nested Sets

Each node stores `left` and `right` boundary values that encode the tree structure.

```sql
CREATE TABLE categories (
  id VARCHAR PRIMARY KEY,
  name VARCHAR NOT NULL,
  lft INTEGER NOT NULL,
  rgt INTEGER NOT NULL
);
-- Descendants of node X: WHERE lft > X.lft AND rgt < X.rgt
```

**Pros:** Single-query descendants and ancestors, efficient reads
**Cons:** Expensive inserts/moves (must renumber boundaries), fragile under concurrent writes

### Option 3: Materialized Path

Each node stores the full path from root as a string (e.g., `electronics/phones/smartphones`).

```sql
CREATE TABLE categories (
  id VARCHAR PRIMARY KEY,
  name VARCHAR NOT NULL,
  path VARCHAR NOT NULL,
  depth INTEGER NOT NULL
);
-- Descendants of "electronics": WHERE path LIKE 'electronics/%'
```

**Pros:** Simple prefix matching for descendants, easy breadcrumbs, readable paths
**Cons:** Path updates cascade to all descendants on move, path length grows with depth

## Decision

We chose **Option 3: Materialized Path** for the category tree implementation.

## Rationale

### Read-Heavy Workload

Category operations in an e-commerce catalog are overwhelmingly read-heavy:

| Operation | Frequency | Notes |
|-----------|-----------|-------|
| Display category tree | Very high | Every page load (cached) |
| Get descendants (product listing) | High | Category page loads |
| Generate breadcrumbs | High | Every product page |
| Reorder categories | Low | Admin action, weekly |
| Move category subtree | Very low | Admin action, monthly |
| Create/delete category | Very low | Admin action, quarterly |

Materialized paths optimize for the high-frequency read operations at the cost of more expensive (but rare) write operations.

### Comparison Matrix

| Criteria | Adjacency List | Nested Sets | Materialized Path |
|----------|---------------|-------------|-------------------|
| Get children | O(n) filter | O(n) filter | O(n) filter |
| Get all descendants | Recursive/CTE | Single query | Prefix LIKE |
| Get ancestors/breadcrumbs | Recursive/CTE | Single query | Split path |
| Insert | O(1) | O(n) renumber | O(1) |
| Move subtree | O(1) | O(n) renumber | O(descendants) |
| Readability | Low | Very low | High |
| Concurrent safety | High | Low | Medium |

### Implementation Details

The path format uses slug segments separated by `/`:

```
electronics                        depth=0
electronics/phones                 depth=1
electronics/phones/smartphones     depth=2
electronics/phones/accessories     depth=2
clothing                           depth=0
clothing/mens                      depth=1
clothing/mens/shirts               depth=2
```

**Breadcrumbs** are generated by splitting the path and resolving each segment:

```typescript
// Path: "electronics/phones/smartphones"
// Breadcrumbs: [
//   { name: "Electronics", path: "electronics" },
//   { name: "Phones", path: "electronics/phones" },
//   { name: "Smartphones", path: "electronics/phones/smartphones" }
// ]
```

See `buildBreadcrumbs()` in the [category model](../src/models/category.ts).

**Descendant queries** use prefix matching:

```typescript
const descendants = categories.filter(c => c.path.startsWith(parentPath + '/'));
```

This is efficient in memory and translates to a `LIKE 'prefix%'` query in SQL with a B-tree index on `path`.

### Move Subtree Operation

Moving a category subtree requires updating the path of all descendants. For a subtree with `d` descendants:

```typescript
// Move "phones" from "electronics" to "mobile-devices"
// Old path prefix: "electronics/phones"
// New path prefix: "mobile-devices/phones"
for (const descendant of descendants) {
  descendant.path = descendant.path.replace(oldPrefix, newPrefix);
  descendant.depth = computeDepth(descendant.path);
}
```

This is O(d) but acceptable given that category moves are extremely rare (monthly at most).

## Consequences

### Positive

- Simple and readable path representation
- Efficient descendant queries via prefix matching
- Easy breadcrumb generation by splitting the path string
- No complex boundary renumbering like nested sets
- Paths are human-readable in database queries and logs

### Negative

- Path updates cascade to all descendants on move operations
- Maximum path length limits tree depth (mitigated by using short slugs)
- No built-in referential integrity for path consistency (enforced in application layer)

### Neutral

- Depth is redundant (can be computed from path) but stored for query convenience
- Sort order is stored per-node, requiring explicit reorder operations

## Path Length Considerations

With an average slug length of 15 characters and `/` separator, the maximum depth before hitting a typical VARCHAR(255) limit is approximately 16 levels. E-commerce categories rarely exceed 4-5 levels, so this is not a practical concern.
